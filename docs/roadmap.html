Overall progress (82 functional requirements)

Where we are

The deterministic core is complete and proven end to end. An RFQ can be parsed (Excel), matched and banded against the catalog, the customer and tier resolved, priced in exact Decimal dong, assembled into a Quote, independently re-checked by the critic (zero diffs), and rendered to bilingual HTML - all offline-tested, 156 tests green.

The cloud plane is live. OSS buckets provisioned, Tablestore knowledge store seeded, and catalog vector + full-text search verified against real ap-southeast-1 (a real result-mapping bug was found and fixed there).

The pipeline runs end to end, live. A Vietnamese RFQ email was parsed by qwen-plus, the customer resolved to dealer tier from the email domain, all three lines matched against the seeded catalog, priced, critic-checked with zero recompute diffs, and rendered - see docs/sample-quote.html.

The human gate is durable. An RFQ posted to /api/rfq is persisted, numbered by an atomic counter, run to pending_approval, and can then be approved by a different process that simply loads it from Tablestore - with a verified hash-chained audit trail. Blocking flags cannot be approved without an audited waiver.

The quote ships. On approval the PDF is rendered by WeasyPrint, stored privately in OSS, fetched back over a V4 presigned URL (verified: HTTP 200, real PDF bytes, diacritics intact), and the bilingual email is dispatched - stub transport writes the same message to the OSS outbox. An RFQ dropped into oss://quotemind-inbox/rfq/ also runs the whole path by itself.

The reviewer can see the reasoning. Every model call, tool call and memory read is traced with real provider token counts, priced, and persisted to OSS; the dashboard renders it as a step-by-step panel beside the quote. Approving a flagged quote raises a waiver modal, and the waiver lands verbatim on the hash-chained audit.

Scans are read, not skipped. Scanned RFQs are rasterised at 200 DPI and read by qwen-vl-ocr. The parser never repairs malformed JSON and never guesses an unreadable quantity - a parser that guesses at a price is worse than one that refuses. This unblocked the last 5 eval cases, so the denominator is now the real 30.

The multi-agent gain is measured, not asserted. All 30 labelled cases now run - including 5 real OCR scans - against a 61-SKU catalog and a single-agent baseline on the same models and inputs. 93% task success and 93% price-exact, against 40% and 40%: a +53 point gap. The baseline got worse as scans were added (48% → 40%) - more documents mean more arithmetic, and it does the arithmetic in the model. It flags a problem 0% of the time. Confidently wrong, silently, is the failure mode the architecture exists to prevent. See eval/reports/ for the numbers, including the ones that did not go our way.

The Alibaba proof is a file you can run. make proof exercises DashScope (chat + 1024-dim embedding), OSS (put → presigned GET → byte-compare → delete) and Tablestore (put → get → delete, diacritics byte-exact) against the real Singapore region: 8/8 checks pass.

It is deployed, and you can click it. The dashboard and /health are live on Function Compute 3.0 in ap-southeast-1. /health reports unverified: [] and substitutions: {} - not a hardcoded banner: the function probes every frozen model id from inside FC, and both lists being empty means each one answered live. Getting there cost four 502s; the honest part is that the first three were guesses, and the fix only came once the function was made to log what FC actually passes it. See api/fc.py and BUILD_LOG.

The autopilot loop closes. A spreadsheet dropped into oss://quotemind-inbox/rfq/ is parsed, matched, priced, critic-checked and parked at the human approval gate by itself - proven live as QM-2026-0006, flagged UNKNOWN_CUSTOMER because a bare file drop carries no sender and the system refuses to guess a pricing tier. Getting there exposed a real bug: both file channels had been decoding every upload as UTF-8, so a spreadsheet became mojibake and a scanned PDF became noise. The eval never caught it because the eval calls the parsers directly - it proved the parsers and never touched the seam joining them to the product.

It remembers, and it plans. On approve or reject the system writes an episodic memory - a bilingual summary, the outcome, the human's own words, and an importance that ranks a rejection above an approval. Before quoting a known customer it recalls the top three, re-ranked by similarity x recency x importance, inside a 1200-token budget, with every memory id in the trace. Memory informs the reviewer and never the price: a retrieved document that could nudge a number would put a similarity search inside the arithmetic path. Non-trivial quotes are planned in AgentScope's PlanNotebook, and the plan is real - its subtasks are closed by the pipeline that actually ran, and one nobody closed still says todo.
